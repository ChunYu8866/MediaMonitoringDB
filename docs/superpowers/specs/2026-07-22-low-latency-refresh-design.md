# 低延遲新聞更新設計

## 目標

在免費 Demo 的既有 Cloudflare Worker 與 GitHub Pages 架構內，讓使用者看到的新聞搜尋與 Google Trends 儘可能接近來源最新狀態，同時避免背景輪詢浪費免費額度。

## 更新策略

- 關鍵字搜尋由 Worker 每次直接查詢可用 RSS 與 Google News；不使用 Cloudflare 共用快取。
- 搜尋完成後，前端每 30 秒以相同關鍵字與時間範圍自動刷新。
- Google Trends 前端每 2 分鐘刷新；Worker 最多快取 60 秒。
- 提供「立即更新」按鈕，並顯示最後成功更新時間與下次更新倒數。
- 瀏覽器分頁隱藏時停止輪詢，重新顯示時立即刷新，避免無效請求。
- 使用者變更關鍵字或時間範圍時，取消舊的更新週期並以新條件立即查詢。

## 資料邊界

- 即時查詢仍只將指定 22 家媒體納入新聞熱度計算。
- Google Trends 附帶的外部新聞只顯示在 Trends 詳情，不納入 22 家媒體統計。
- 媒體 RSS 或 Google News 尚未發布的文章無法提前取得；介面不得宣稱秒級即時。
- Worker 失敗時才改讀 GitHub Pages 最後快照，並明確標示為快照。

## 錯誤處理

- 自動刷新失敗時保留上一份成功結果，不清空畫面。
- 顯示刷新失敗提示與最後成功時間；下一輪仍會重試。
- 手動刷新與自動刷新共用同一請求狀態，避免同時發出重複請求。

## 驗證

- Worker 測試確認搜尋回應不再宣告可共用快取，Trends 快取不超過 60 秒。
- 前端測試確認輪詢間隔、分頁隱藏暫停與重新顯示立即刷新。
- 型別檢查、正式建置與既有測試全部通過。
- 部署後驗證 Worker 回應標頭、正式站新 UI、GitHub Pages Action 與實際更新時間。

## 不在本次範圍

- 付費新聞 API、全文爬取、WebSocket、D1/KV 常駐新聞資料庫。
- 將 GitHub Actions 排程縮短到五分鐘；其角色維持低頻快照備援。
